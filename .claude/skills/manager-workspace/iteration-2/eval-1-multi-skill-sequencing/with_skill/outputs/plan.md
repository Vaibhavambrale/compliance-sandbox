## Summary
Three specialist skills (Frontend, Backend, Database) have proposed a Trends feature: a new `/trends` page with a Recharts line chart, a new `GET /api/trends` route, a new `lib/api/trends.ts` client wrapper, a Sidebar nav entry, a new `trend_bucket text` column on `test_runs`, and a `created_at DESC` index. The `trends` route does not call any LLM and does not stream SSE, so the Edge runtime requirement applies only if the route would otherwise time out — it is not an LLM/SSE route but using Edge is safe and consistent with the rest of the codebase. No active bugs (BUG 1–8) are directly touched by this feature set, but BUG 6 (`readiness_score` is currently always `NULL` because Layer 2 is absent) means the Trends chart will render a flat or near-empty line for most historical runs — this must be called out to the user before work begins.

## Conflicts detected
- [Database] wants to add a `trend_bucket text` column to `test_runs`, but [Frontend] and [Backend] never reference `trend_bucket` in their proposals — Frontend renders `readiness_score` over time, Backend returns `{ id, created_at, readiness_score, readiness_tier, model_name }`. **Winner: Frontend + Backend.** Why: the column is unused by either consumer skill; adding dead schema without a migration plan and without any code that writes or reads it layers unreviewable debt onto a table that already has unresolved bugs (BUG 2 touches `test_runs` inserts). The `trend_bucket` column should be deferred or the Database skill must clarify which code path populates it.
- [Backend] proposes using the service role Supabase client in `app/api/trends/route.ts`. The route performs a simple read (`SELECT id, created_at, readiness_score, readiness_tier, model_name FROM test_runs ORDER BY created_at DESC LIMIT 30`) with no writes and no bypass of RLS being strictly necessary. **Winner: Hard Rule 3 (enforcement, not a true inter-skill conflict).** Why: using the service role key for a read-only route that has no reason to bypass RLS is higher-risk than needed — if this route is ever called client-side with a misconfigured CORS policy the key is exposed. The route should use the anon client unless RLS is confirmed to block the read, in which case RLS must be relaxed for this query explicitly. This is flagged as a Hard Rule 3 risk below.

## Hard-rule violations
- **Rule 3 violation risk — `SUPABASE_SERVICE_ROLE_KEY` in `app/api/trends/route.ts` (Backend proposal).** Backend explicitly proposes using the service role client. `app/api/trends/route.ts` is a server-side API route, so the key does not leak to the browser bundle directly. However, using the service role key in a route that only needs a simple SELECT is an unnecessary escalation of privilege. CLAUDE.md states the service role key must only be used server-side (satisfied) but the project pattern is to use it only where RLS would otherwise block a legitimate operation. No RLS policy exists for `test_runs` (confirmed in CLAUDE.md infrastructure gaps: "No Supabase RLS policies in repo"). Until RLS is off, the anon client can already read `test_runs`, making the service role key gratuitous here. This is a **conditional violation**: if the Editor uses the service role client without documenting why the anon key is insufficient, it violates the principle behind Rule 3. The plan must specify: use anon client; if RLS is later turned on for `test_runs`, revisit. This is **not a hard NO-GO** by itself since the key stays server-side, but must be corrected before implementation.

## Bottleneck warnings
- **BUG 6 — `readiness_score` is always NULL for most existing runs.** Layer 2 (Capability Benchmarking) is entirely absent; `readiness_score` is set to `compliance_score` in `app/api/test/run/route.ts:~311` rather than computed from both layers. The Trends chart will render correctly for new runs but historical runs may show `NULL` values that Recharts will render as gaps. Frontend skill must handle `null` values in the chart data gracefully (filter them out or show a `no data` state) rather than crashing or rendering misleading zeroes.
- **No RLS on `test_runs`.** The new `/api/trends` endpoint reads from `test_runs`. Since no RLS is in place, the anon key can already read all rows — this is acceptable for now but must be noted. When RLS is eventually enabled (Phase 1, item 7 in CLAUDE.md remaining work), this route will need a policy to allow reads.
- **No authentication / user isolation.** `test_runs.user_id` is schema-only; the Trends page will show all users' runs aggregated. For the IEEE demo this is fine, but it must not be presented as per-user analytics.
- **Supabase free-tier pausing.** No cron keep-alive exists. If the database pauses during the demo window, the Trends page (like all other DB-dependent pages) will fail silently. Set up a cron-job.org ping before the presentation date.
- **Readiness-tier logic is duplicated 3×.** Adding the Trends page introduces a fourth consumer of `readiness_tier` display logic. The plan must not add a fourth inline copy — use the same color/label pattern already established in the existing pages, and flag this as debt to be resolved in `lib/scoring.ts` (Phase 2, item 2 in CLAUDE.md remaining work).
- **`trend_bucket` column conflict is unresolved.** If the Database skill's column is implemented without a writer, it becomes permanent dead weight on the `test_runs` table. This is particularly risky given the existing BUG 2 (`model_provider` NOT NULL band-aid) which shows that unplanned schema additions to `test_runs` have already caused insert failures. Do not merge the column until a writer is defined.
